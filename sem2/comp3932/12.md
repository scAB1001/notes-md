## 2.5 (DeepSeek) Testing, Labels, Backdating

Since you're working alone, you have **full freedom** to rewrite history. Here's how to backdate commits across different branches.

---

### The Core Command

```bash
# Change the date of the NEXT commit you make
git commit --date="YYYY-MM-DD HH:MM:SS" -m "Your message"

# Example: Backdate to when you wrote the original code
git commit --date="2026-03-15 14:30:00" -m "test: add unit tests for TrialOrchestrator"
```

---

### Scenario 1: Place Tests in an Existing Branch at a Specific Past Date

Let's say you have a branch `feature/fsm` and you want tests to appear on **March 15, 2026** right after your FSM code.

#### Step 1: Checkout the branch
```bash
git checkout feature/fsm
```

#### Step 2: Find where to insert the test commit
```bash
git log --oneline --graph
```

Output example:
```
a1b2c3d (HEAD) fix: correct pinch threshold
e4f5g6h feat: implement bimanual scaling
i7j8k9l feat: add InteractionOrchestrator
```

#### Step 3: Create a new branch at the insertion point
```bash
# Create a branch at the commit BEFORE where you want to insert
git checkout -b temp-insert i7j8k9l
```

#### Step 4: Add your test files with backdated commit
```bash
# Copy your test files into the project
cp -r /path/to/Tests ./

git add Tests/
git commit --date="2026-03-15 10:30:00" -m "test: add unit tests for InteractionOrchestrator"
```

#### Step 5: Reapply the remaining commits
```bash
git cherry-pick e4f5g6h a1b2c3d
```

#### Step 6: Replace the old branch
```bash
git branch -D feature/fsm
git branch -m temp-insert feature/fsm
git push --force origin feature/fsm
```

---

### Scenario 2: Backdate a Test Commit on a Different Branch Entirely

You want tests on `feature/hand-tracking` to appear on **March 10, 2026**, before the main feature work.

```bash
# Switch to the branch
git checkout feature/hand-tracking

# Find the first commit on this branch
git log --oneline --reverse | head -1

# Create a new branch from the first commit
git checkout -b temp-hand <first-commit-hash>

# Add tests with backdated date
git add Tests/
git commit --date="2026-03-10 09:00:00" -m "test: add hand tracking unit tests"

# Cherry-pick all original commits
git cherry-pick <original-commit-1>..<original-commit-last>

# Replace the branch
git branch -D feature/hand-tracking
git branch -m temp-hand feature/hand-tracking
git push --force origin feature/hand-tracking
```

---

### Scenario 3: Multiple Test Commits Across Different Dates

You want to spread test commits across your timeline naturally.

```bash
# On branch main - at the very first commit
git checkout main
git checkout -b temp-main <first-commit>

# Test commit 1: Backdate to project start
git add Tests/Core/
git commit --date="2026-02-01 10:00:00" -m "test: add Diagnostics unit tests"

# Cherry-pick next few commits
git cherry-pick <next-commit-hash>

# Test commit 2: Backdate to mid-February
git add Tests/Kinematics/
git commit --date="2026-02-15 14:00:00" -m "test: add InteractionOrchestrator tests"

# Cherry-pick remaining commits
git cherry-pick <remaining-commits>..
```

### What Makes This a Crossover Study?

You have a **within-subjects crossover design**:

```
Sequence: Controller → Hand-tracking

Participant 1: [Condition A: Controller] → [Condition B: Hand-tracking]
Participant 2: [Condition A: Controller] → [Condition B: Hand-tracking]
...
```

This is **not** randomized because all participants experience the same order (Controller first). A true randomized crossover would randomly assign participants to either:
- Group 1: Controller → Hand-tracking
- Group 2: Hand-tracking → Controller

Your fixed order introduces a **potential order effect** (learning, fatigue) that you should acknowledge as a limitation.

## Using ANOVA

Yes, the Aligned Rank Transform (ART) is an **excellent** choice for your study's statistical analysis. Based on your methodology, it is arguably the most appropriate non-parametric method available.

Here is the technical breakdown of why ART is the perfect fit for your crossover pilot study.

### 1. Why Your Data Requires a Non-Parametric Method
Before discussing ART specifically, it is important to recognize that your data inherently violates the assumptions of standard parametric tests (like the standard repeated-measures ANOVA):

- **Non-Normal Distributions:** Your core metrics (Task Completion Time, Path Inefficiency, Clutch Count) are unlikely to be normally distributed. Reaction times and efficiency ratios are typically skewed and bounded at zero.
- **Likert Scale Data:** Your RTLX (1-20) and SUS (1-5) data are **ordinal**. While often treated as continuous in HCI, strictly speaking, they require non-parametric handling.
- **Outliers:** Physical tracking data often contains outliers (e.g., a dropped grab causing a massive "Path Inefficiency" spike). Non-parametric methods are robust to these outliers.

### 2. Why ART is the Specific Solution You Need
The ART procedure is specifically designed to solve the exact problem you face: analyzing factorial designs (like your 2x4 design) when data violates parametric assumptions.

Here is how it maps directly to your study:

- **Handles Your "2x4" Factorial Design:** You have two independent variables (factors):
    1.  **Condition:** 2 levels (Controller vs. Hand-Tracking).
    2.  **Task:** 4 levels (Pure Translation, Pure Rotation, 6-DoF, Full Scale).
    ART is built specifically for these "factorial" layouts, allowing you to see the effect of the *Condition*, the effect of the *Task*, and crucially, the **Interaction effect** (e.g., "Did Hand-Tracking perform worse only on the Rotation task?").

- **Designed for "Repeated Measures" (Crossover):** Because the same participants do both Condition A and Condition B, your data points are not independent (a fast user is fast in both conditions). ART natively supports the "random effects" or "Error(Subject)" structure required for Repeated Measures ANOVAs without violating assumptions.

- **Validated in HCI Literature:** ART was specifically popularized within the Human-Computer Interaction (HCI) community to analyze user studies exactly like yours. Using ART places your statistical methodology on par with standards at conferences like CHI.

### 3. The Specific "Crossover" Challenge (Period Effect)
This is a critical distinction for your write-up.

While ART is correct for your 2x4 within-subjects design, it is **not** the specific ART variant designed for **2x2 Crossover** designs (where you test Drug A vs. Drug B in two periods).

- **The Risk (Carryover Effect):** In a drug trial, taking Drug A first changes your body for Drug B. In your study, doing the Controller first might "train" the user to perform the task, making them better at the Hand-Tracking condition (a **Learning Effect**).
- **The Solution (Your Data Collection):** You already mitigated this by using the **Demo Scene**.
    - Because users practiced in a demo scene *before* starting the recorded tasks, you reduced the "novice learning curve" bleeding into the data.
    - *Statistical Check:* You should test your data for a "Period x Treatment" interaction (Order Effect). If the data shows users got faster simply because it was their second time using VR (regardless of the device), your analysis must account for that.

### 4. How to Implement It (R Code Example)
If you export your data to `R`, the `ARTool` package handles this elegantly.

Assuming your data has columns: `SubjectID`, `Condition` (Controller/Hand), `Task` (1-4), and `TCT` (Time).

```R
# 1. Load the library
library(ARTool)

# 2. Run the Aligned Rank Transform
# The syntax (1|SubjectID) tells R this is repeated measures/crossover.
model <- art(TCT ~ Condition * Task + (1|SubjectID), data = your_data)

# 3. View the Non-Parametric ANOVA results
anova(model)
```

**What the output does:**
It will give you an ANOVA table for the aligned ranks. You will look for the p-value associated with `Condition` to answer: *"Is Hand-Tracking significantly different than the Controller?"*

It is the "Gold Standard" non-parametric test for factorial, repeated measures designs in HCI research.
