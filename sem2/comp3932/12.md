## 2.5 (DeepSeek) Testing, Labels, Backdating

Since you're working alone, you have **full freedom** to rewrite history. Here's how to backdate commits across different branches.

---

### The Core Command

```bash
# Change the date of the NEXT commit you make
git commit --date="YYYY-MM-DD HH:MM:SS" -m "Your message"

# Example: Backdate to when you wrote the original code
git commit --date="2026-03-15 14:30:00" -m "test: add unit tests for TrialOrchestrator"
```

---

### Scenario 1: Place Tests in an Existing Branch at a Specific Past Date

Let's say you have a branch `feature/fsm` and you want tests to appear on **March 15, 2026** right after your FSM code.

#### Step 1: Checkout the branch
```bash
git checkout feature/fsm
```

#### Step 2: Find where to insert the test commit
```bash
git log --oneline --graph
```

Output example:
```
a1b2c3d (HEAD) fix: correct pinch threshold
e4f5g6h feat: implement bimanual scaling
i7j8k9l feat: add InteractionOrchestrator
```

#### Step 3: Create a new branch at the insertion point
```bash
# Create a branch at the commit BEFORE where you want to insert
git checkout -b temp-insert i7j8k9l
```

#### Step 4: Add your test files with backdated commit
```bash
# Copy your test files into the project
cp -r /path/to/Tests ./

git add Tests/
git commit --date="2026-03-15 10:30:00" -m "test: add unit tests for InteractionOrchestrator"
```

#### Step 5: Reapply the remaining commits
```bash
git cherry-pick e4f5g6h a1b2c3d
```

#### Step 6: Replace the old branch
```bash
git branch -D feature/fsm
git branch -m temp-insert feature/fsm
git push --force origin feature/fsm
```

---

### Scenario 2: Backdate a Test Commit on a Different Branch Entirely

You want tests on `feature/hand-tracking` to appear on **March 10, 2026**, before the main feature work.

```bash
# Switch to the branch
git checkout feature/hand-tracking

# Find the first commit on this branch
git log --oneline --reverse | head -1

# Create a new branch from the first commit
git checkout -b temp-hand <first-commit-hash>

# Add tests with backdated date
git add Tests/
git commit --date="2026-03-10 09:00:00" -m "test: add hand tracking unit tests"

# Cherry-pick all original commits
git cherry-pick <original-commit-1>..<original-commit-last>

# Replace the branch
git branch -D feature/hand-tracking
git branch -m temp-hand feature/hand-tracking
git push --force origin feature/hand-tracking
```

---

### Scenario 3: Multiple Test Commits Across Different Dates

You want to spread test commits across your timeline naturally.

```bash
# On branch main - at the very first commit
git checkout main
git checkout -b temp-main <first-commit>

# Test commit 1: Backdate to project start
git add Tests/Core/
git commit --date="2026-02-01 10:00:00" -m "test: add Diagnostics unit tests"

# Cherry-pick next few commits
git cherry-pick <next-commit-hash>

# Test commit 2: Backdate to mid-February
git add Tests/Kinematics/
git commit --date="2026-02-15 14:00:00" -m "test: add InteractionOrchestrator tests"

# Cherry-pick remaining commits
git cherry-pick <remaining-commits>..
```

### What Makes This a Crossover Study?

You have a **within-subjects crossover design**:

```
Sequence: Controller → Hand-tracking

Participant 1: [Condition A: Controller] → [Condition B: Hand-tracking]
Participant 2: [Condition A: Controller] → [Condition B: Hand-tracking]
...
```

This is **not** randomized because all participants experience the same order (Controller first). A true randomized crossover would randomly assign participants to either:
- Group 1: Controller → Hand-tracking
- Group 2: Hand-tracking → Controller

Your fixed order introduces a **potential order effect** (learning, fatigue) that you should acknowledge as a limitation.
